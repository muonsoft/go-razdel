# Execution workflow

## Цикл выполнения одной задачи

1. Выбрать задачу `TXXX` из `control/tracker.md`.
2. Подтвердить upstream reference (тест/пример/код).
3. Внести минимальное изменение в Go-код.
4. Добавить/обновить тесты parity для этого изменения.
5. Запустить обязательные проверки:
   - `go test ./...`
   - `go vet ./...`
   - `golangci-lint run ./...`
6. Зафиксировать статус в `control/tracker.md`.
7. Если есть расхождение с upstream — задокументировать в notes/README.

## Правило гранулярности

- Один PR-коммит = одна логическая подзадача.
- Не переносить много rule-групп в одном изменении.
- Сложные изменения дробить на pre-step (инфраструктура тестов) и behavior-step.

## Порядок выполнения фаз

1. `T001-T003`: инфраструктура и контракты.
2. `T004-T010`: токенизация.
3. `T011-T017`: сегментация предложений.
4. `T018`: differential/fuzz/bench и стабилизация.

## Управление рисками

- Любой флейк в интеграционных тестах устранять через deterministic sampling.
- Изменения в testdata фиксировать с метаданными (источник, seed, SHA).
- Нельзя менять `third_party/razdel` в feature-задачах.

## Definition of done (на уровне задачи)

Задача считается завершенной, если:

1. Реализация покрывает конкретный behavior slice.
2. Есть тесты, ссылающиеся на upstream.
3. Все quality gates проходят.
4. Статус в `control/tracker.md` обновлен.
