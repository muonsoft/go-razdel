# go-razdel contracts (v0)

Этот документ фиксирует верхнеуровневые контрактные требования для первой версии библиотеки.

## Scope

- Контракты относятся к публичным API токенизации и сегментации предложений.
- Цель этапа: предсказуемое поведение и совместимость с upstream (`third_party/razdel`) на уровне результатов.
- Документ задает требования, а не внутреннюю реализацию.

## Public API contract

Минимальный публичный API v0:

- `Tokenize(text string) []Token`
- `Sentenize(text string) []Sentence`

Ожидаемые структуры результата:

- `Span` содержит `Start` и `End`.
- `Token` и `Sentence` содержат `Span` и `Text`.

Допустимы внутренние изменения реализации, если не нарушены требования ниже.

## Offsets contract (Variant A)

Принят вариант A:

- `Start` и `End` измеряются в UTF-8 байтах исходной строки.
- `Start` inclusive, `End` exclusive.
- Для каждого элемента верно:
  - `0 <= Start <= End <= len(text)`
  - `Text == text[Start:End]`

Это базовая инварианта API. Ее нарушение считается критической ошибкой контракта.

## Behavioral contract

- `Tokenize` и `Sentenize` детерминированы: одинаковый вход -> одинаковый выход.
- Функции не имеют побочных эффектов.
- Для пустой строки возвращается пустой срез результата.
- Поведение должно быть максимально совместимо с upstream.
- Известные расхождения фиксируются в документации с примерами и обоснованием.

## Compatibility contract with upstream

- Upstream в `third_party/razdel` используется как эталон поведения.
- Каждый реализуемый кусок поведения подтверждается:
  - ссылкой на upstream-тест/пример, или
  - локальным golden-тестом, эквивалентным upstream-кейсу.
- Обновление submodule не выполняется в рамках feature-задач и идет отдельным изменением.

## Error-handling contract

- Публичные функции v0 не возвращают `error`.
- Публичные функции не должны panic на корректном UTF-8 вводе.
- Поведение на некорректном UTF-8 должно быть явно задокументировано до публикации стабильного релиза.

## Performance contract (non-functional)

- Временная сложность: линейная относительно длины входной строки.
- Не допускаются неоправданные аллокации на каждый символ.
- Для каждой крупной итерации желательны benchmark-тесты на:
  - коротких строках;
  - длинных строках;
  - Unicode-heavy текстах;
  - текстах с плотной пунктуацией.

## Contract change policy

- Любое изменение этих контрактов требует отдельного PR/коммита и явного описания миграции.
- Изменение единицы смещений (`byte` -> что-то другое) запрещено в пределах v0/v1 без major-перехода.
