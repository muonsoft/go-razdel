# AGENTS Guidelines

## Scope

Этот репозиторий предназначен для поэтапного переноса `natasha/razdel` на Go с контролем поведенческой совместимости.

## Working Rules

0. **Перед началом любой задачи** убедиться, что submodule `third_party/razdel` развёрнут (не пустой каталог): выполнить `git submodule update --init --recursive` при необходимости и проверить, что в `third_party/razdel` есть файлы (например `razdel/` или корневой `setup.py` upstream). Без этого upstream-ссылки из задач недоступны и parity нельзя проверить по эталону.
1. Не изменять `third_party/razdel` без явной задачи на обновление upstream.
2. Любое обновление submodule делать отдельным коммитом с указанием старого и нового commit SHA.
3. При реализации ориентироваться на behavioral parity с upstream-реализацией.
4. Делать небольшие, проверяемые инкременты вместо крупных одномоментных изменений.
5. Спорные решения (границы токенизации, Unicode edge cases, формат API) фиксировать в `README.md` или отдельных notes-файлах.
6. На bootstrap-этапе не переносить алгоритмы из Python в Go.

## Linear: статусы задач

Если в описании задачи, ветке, коммите или PR есть идентификатор Linear (например `TEAM-123`, `LIN-456`, ссылка на `linear.app`), агент **обязан** синхронизировать статус issue с ходом работы через доступные Linear-инструменты (MCP).

### Когда обновлять

1. **Старт работы** — сразу после того, как понятно, что именно эта issue в фокусе (взята в работу): перевести в статус «в работе» для команды (часто *In Progress* или аналог). Не менять статус, если issue только упомянута как контекст и работа ведётся по другой задаче.
2. **Готово к ревью** — после `git push`, когда открыт или обновлён PR и локальные проверки из раздела *Validation Expectations* пройдены (или явно зафиксировано, что они недоступны в среде): перевести в статус ревью, если в команде он есть (*In Review*, *Pending Review* и т.п.).
3. **Завершение** — когда критерии приёмки выполнены и PR готов к слиянию (или по явной формулировке в задаче «закрыть issue»): перевести в финальный статус (*Done*, *Completed*, *Canceled* — только если задача отменена).

Порядок статусов в разных командах Linear различается: перед первым изменением **получить актуальные имена** через API/workspace (например, список статусов команды) и подставлять **точное имя** статуса, как в Linear.

### Как не ошибиться

- Обновлять **только** привязанную к текущей сессии issue; не массово менять чужие задачи.
- Если идентификатор неоднозначен или несколько issue — уточнить по описанию задачи или спросить пользователя; не гадать.
- Кратко отразить прогресс в **комментарии** к issue при смене статуса (что сделано, ссылка на ветку/PR), если это уместно и не дублирует автоматику платформы.
- Если Linear недоступен (нет MCP, ошибка API) — продолжить работу в репозитории и зафиксировать в ответе пользователю, что статус не обновлён.

## Validation Expectations

- Перед `go test` на задаче, зависящей от upstream, submodule должен быть инициализирован (см. правило 0).
- Для каждого изменения добавлять или обновлять тесты.
- Проверять локально минимальный набор команд:
  - `go test ./...`
  - `go vet ./...`
  - `golangci-lint run ./...`

## Definition of Ready for Implementation Tasks

- Есть четкая ссылка на upstream-поведение (файл/пример/тест).
- Изменение ограничено конкретной, измеримой целью.
- Критерии приемки выражены через тесты или воспроизводимые примеры.
